         Action Reference: A Deterministic Identifier for Agent Actions
                      draft-giskard-action-ref-00

Abstract

   This document defines "action_ref", a deterministic, content-
   addressed identifier for autonomous agent actions.  Any party holding
   the four preimage fields (agent_id, action_type, scope, timestamp)
   can independently compute and verify the identifier without trusting
   the emitting system.  The document specifies the derivation algorithm,
   canonical receipt envelope, optional fields for revocation and policy
   rotation auditability, and a composability model with existing
   exactly-once execution guarantees.

1.  Introduction

   Autonomous agents operating across distributed systems need a way to
   identify individual actions that is independent of any single
   system's infrastructure.  Existing audit trail mechanisms rely on
   identifiers assigned by the recording system — identifiers that
   require trusting that system's integrity, availability, and continued
   operation.

   "action_ref" is a content-addressed identifier derived exclusively
   from the semantic content of the action itself.  Its key properties:

   o  Deterministic: the same four inputs always produce the same
      identifier.

   o  Independent: any party with the preimage can verify the
      identifier without contacting the emitter.

   o  Composable: the same identifier links records across independently
      operated systems (audit trails, idempotency guards, negotiation
      artifacts) without requiring shared infrastructure.

   o  Durable: the identifier remains verifiable after the emitting
      system is offline, has rotated keys, or no longer exists.

   This document defines the derivation algorithm, the canonical receipt
   envelope, optional fields for post-rotation auditability, and the
   composability model with exactly-once execution guarantees.

   The specification has been implemented in production systems anchoring
   agent action trails on Arbitrum [ARGENTUM-CORE], validated against 53
   conformance vectors across 5 implementation languages, and recognized
   as a referenced contribution in the AEOESS Agent Trust and
   Accountability Cross-Extension Framework Matrix v0.3.3 [CTEF-MATRIX]
   (row #2, "urn:mycelium:trail", CONFIRMED).


2.  Terminology

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
   "OPTIONAL" in this document are to be interpreted as described in
   BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all
   capitals, as shown here.

   action_ref:     The deterministic identifier defined by this document.
                   A SHA-256 hex string derived from the JCS
                   serialization of the four preimage fields.

   preimage:       The four-field JSON object from which action_ref is
                   derived: {agent_id, action_type, scope, timestamp}.

   emitter:        The system or agent that computes and publishes an
                   action_ref.

   verifier:       Any party that recomputes action_ref from preimage
                   fields to confirm integrity.

   claim time:     The moment an action is declared before execution.
                   Distinct from execution time, which may be later.

   TrailRecord:    An immutable, on-chain-anchored record of an agent
                   action containing the action_ref and preimage fields.


3.  Action Reference Derivation

3.1.  Preimage Fields

   The action_ref preimage is a JSON object with exactly four fields:

   agent_id (string, REQUIRED):
      A stable identifier for the acting agent.  MAY be a DID, a
      registered username, or an opaque string.  MUST be non-empty.
      MUST be consistent across all parties deriving the same
      action_ref for the same action.

   action_type (string, REQUIRED):
      A semantic label for what the agent did.  Examples:
      "code.execute", "payment.send", "oracle.signal".  MUST be
      non-empty.

   scope (string, REQUIRED):
      The declared authorization boundary — what the agent was
      permitted to do, not what it did.  MUST be non-empty.  See
      Section 6 for conventions.  Pass the empty string "" only if
      scope is explicitly not applicable; implementations MUST treat
      "" and absent scope as distinct values.

   timestamp (string, REQUIRED):
      The moment the action was claimed (before execution), expressed
      as RFC 3339 UTC with exactly 3 millisecond digits.  The trailing
      "Z" suffix is REQUIRED.  Format: "YYYY-MM-DDTHH:MM:SS.mmmZ".
      See Section 3.4 for rationale and conversion rules.

3.2.  Serialization

   The four preimage fields MUST be serialized using JSON
   Canonicalization Scheme (JCS) as defined in [RFC8785]:

   o  Keys in lexicographic Unicode code point order:
      "action_type", "agent_id", "scope", "timestamp"

   o  No whitespace between tokens.

   o  UTF-8 encoded.

   o  Values are JSON strings with standard JSON escaping.

   Implementations operating within the safe band (ASCII-only field
   values, conformant RFC 3339 timestamps, no surrogate-pair Unicode)
   MAY use a standard JSON serializer with sorted keys.  Implementations
   outside this band MUST use an RFC 8785-compliant library.

3.3.  Hash Function

   action_ref = lowercase-hex( SHA-256( JCS(preimage) ) )

   The output is a 64-character lowercase hexadecimal string.

3.4.  Timestamp Format

   The timestamp field represents the moment the action was claimed,
   before execution.  Conformant format:

      YYYY-MM-DDTHH:MM:SS.mmmZ

   Requirements:

   o  Timezone: "Z" suffix only.  "+00:00" or any numeric offset is
      non-conformant and will produce a different digest.

   o  Fractional precision: exactly 3 digits (milliseconds).  No
      trailing zero suppression.  "10:00:00.1Z" is non-conformant.

   o  The format admits exactly one valid byte sequence per instant.

   A verifier receiving a non-conformant timestamp will compute a
   different digest and correctly reject the receipt.  An emitter
   generating non-conformant timestamps produces an unverifiable
   receipt.

   Implementations with epoch-millisecond integer representations MUST
   convert to the conformant string format before hashing.  Multiplying
   by 1000 and applying the format function is lossless.


4.  Canonical Receipt Envelope

4.1.  Required Fields

   Implementations that emit a receipt referencing this specification
   SHOULD include the following fields:

      {
        "packet_version": "1.0",
        "action_ref": "<64-char sha256 hex>",
        "hash_algo": "sha256",
        "preimage_format": "jcs-rfc8785-v1",
        "preimage": {
          "agent_id": "...",
          "action_type": "...",
          "scope": "...",
          "timestamp": "2026-05-26T10:00:00.000Z"
        }
      }

   packet_version:     Forward-compatibility anchor.  Verifiers SHOULD
                       explicitly reject unknown versions.

   hash_algo:          Makes receipts self-describing.  Enables future
                       algorithm agility without breaking receipts
                       issued under prior versions.

   preimage_format:    Unambiguously identifies the serialization.
                       "jcs-rfc8785-v1" corresponds to this document.
                       A verifier can recompute action_ref from the
                       preimage fields using RFC 8785 without trusting
                       the emitter.

4.2.  Optional Fields for Rotation Auditability

   The base envelope records the state at claim time.  It does not
   record whether that state was still valid when the action was
   admitted for execution.  The following optional fields close this
   gap:

   policy_version (string, OPTIONAL):
      Identifies the version of the governing policy in force when the
      action was admitted for execution.  A verifier replaying the
      receipt after a policy rotation can use this field to establish
      that the admitted policy was not superseded before execution.

   authority_verified_at_ms (integer, OPTIONAL):
      Unix timestamp in milliseconds at which the acting agent's
      delegation authority was verified at issuance.  Provides the
      issuance-side anchor for year-5 supervisory re-verification.

   revocation_check_at_ms (integer, OPTIONAL):
      Unix timestamp in milliseconds of the last non-revocation check
      performed before execution.  A receipt without this field cannot
      prove credentials were valid immediately before admission — only
      that they were valid at claim time.  With this field, a verifier
      can establish a maximum window of credential exposure.

      Millisecond precision is REQUIRED because credential checks in
      live multi-agent systems occur at sub-second granularity, and the
      check-to-execution gap is often under one second.

   negotiation_ref (string, OPTIONAL):
      SHA-256 hex pointer to the negotiation artifact (capability-
      grant, covenant, or agreement) that authorized this action.
      Derived as SHA-256(JCS(negotiation_artifact)).  Does NOT enter
      the action_ref preimage: changing or removing negotiation_ref
      does not change action_ref.

   A verifier requiring post-rotation auditability SHOULD treat a
   receipt missing authority_verified_at_ms or revocation_check_at_ms
   as unauditable for the rotation window — not as invalid.


5.  Composability with Exactly-Once Execution

5.1.  Layer Model

   action_ref is designed to compose with, not replace, existing
   execution guarantees.  Two complementary layers:

   Layer 2 — Exactly-once execution (e.g., idempotency guards):
      Proves: this action was claimed before execution; a duplicate
      retry with the same key returned SKIP.
      Does not prove: the outcome is tamper-evident after the fact.

   Layer 4 — Tamper-evident audit trail (e.g., Mycelium TrailRecord):
      Proves: this action occurred, with this hash, anchored at this
      block; the record has not been modified since anchor time.
      Does not prove: the action occurred exactly once.

   A verifier finding both a TrailRecord AND a COMMITTED claim for the
   same action_ref can assert: the action ran exactly once AND the
   outcome is tamper-evident.  Neither layer requires the other;
   implementations MAY adopt either independently.

5.2.  Canonical Linking Key

   The same action_ref MUST be computable from any system holding the
   four preimage fields.  This property enables cross-system
   verification without shared infrastructure.  Known convergences:

   o  Mycelium TrailRecord: preimage fields published in each record.
      [ARGENTUM-CORE]

   o  SafeAgent execution claims: joint spec at [ARGENTUM-CORE-ISSUE7].
      The action_ref maps to SafeAgent's idempotency_key (caller-
      computed).

   o  CrewAI idempotency key: key derivation converges on the same
      primitive from the retry-deduplication direction.
      [CREWAI-PR5822]

   o  Nobulex covenant receipts: action_type as semantic label +
      timestamp + agent_id + scope.

   o  NEXUS oracle receipts: implements canonical envelope v1.0.


6.  Scope Conventions

   scope is a free-form non-empty string with no closed enumeration.
   Any value is valid as long as it is non-empty and consistent across
   all parties deriving the same action_ref.

   RECOMMENDED convention (non-normative): namespace-prefix with the
   emitter identifier using "<emitter>:<scope>".

   Examples:
      "algovoi:compliance_screen"
      "vauban:stark_settlement"
      "agent_os:committed_claim"
      "aura:reputation_observe"

   Rationale: different emitters may independently choose the same
   scope string ("audit", "settlement", "signal") with semantically
   distinct meanings.  Prefixing avoids collisions when trails from
   multiple emitters are aggregated by a third party.

   A verifier MUST NOT reject a trail solely because its scope lacks
   a namespace prefix.


7.  Conformance

   A conformant emitter:

   o  Produces action_ref as lowercase-hex(SHA-256(JCS(preimage))).
   o  Uses the timestamp format defined in Section 3.4.
   o  Includes packet_version, hash_algo, and preimage_format in
      receipts.

   A conformant verifier:

   o  Recomputes action_ref from the four preimage fields.
   o  Rejects receipts where the recomputed value does not match
      the claimed action_ref.
   o  Does not require the emitting system to be online or trusted.

   Conformance test vectors are provided in Appendix A.


8.  Security Considerations

   Collision resistance: action_ref relies on the second-preimage
   resistance of SHA-256.  An attacker who can find two distinct
   preimages that produce the same action_ref can forge a receipt.
   SHA-256 is currently considered secure against this attack.

   Timestamp manipulation: if an emitter controls the timestamp field,
   it can produce a different action_ref for the same semantic action
   by varying the millisecond component.  Systems requiring
   non-repudiation SHOULD anchor the timestamp to an external time
   source and include authority_verified_at_ms.

   Scope ambiguity: without namespace prefixing, scope collisions
   across emitters can cause cross-system verification failures.
   Implementations SHOULD follow the conventions in Section 6.

   Key rotation: action_ref is independent of cryptographic keys.
   Key rotation by the emitting agent does not invalidate previously
   issued receipts.  The revocation_check_at_ms field provides
   auditability for the credential validity window.

Appendix B.  Field Mapping for Existing Systems

   DashClaw [ucsandman/DashClaw]:

      Joint spec field   | DashClaw field
      -------------------|-----------------------------
      agent_id           | agent_id
      action_type        | action_type
      scope              | authorization_scope
      action_ref         | idempotency_key (caller-computed)
      action_id          | DashClaw's local evidence row ID

   Callers compute action_ref externally and pass it as idempotency_key.
   DashClaw consumes it opaquely.  No runtime coupling required.
